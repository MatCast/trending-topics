# AI Architect Lessons & Learnings

This file acts as the continuous learning memory base for the Architect agent.
Whenever a sub-agent fails a task, struggles with a specific dependency, or when a workflow strategy is proven ineffective, the Architect must document the lesson here.

The Architect must read this file at the start of every session to avoid repeating mistakes.

## Lessons

1. **Direct Implementation vs Sub-agents**: For tasks involving complex orchestration (like Docker Compose across multiple services), the user may prefer direct implementation by the Architect to ensure consistency and avoid environment/CLI overhead. Use sub-agents for more isolated, atomic tasks.
2. **Firestore Composite Indexes**: Range filters on multiple fields (e.g., `expires_at > now` and `order_by("created_at")`) require composite indexes. If indexes cannot be created, filter by the primary criteria in Firestore and apply supplementary filters (like expiration checks) in memory after streaming results.
3. **Pydantic Validation Sensitivity**: When returning lists of objects through FastAPI, ensure every object contains all required fields (e.g., `id`). In-place dictionary updates in backend clients can ensure returned lists are valid for Pydantic response models.
4. **Frontend validation visibility**: Always provide immediate visual feedback (e.g., `input-error` class and error labels) when input validation fails, especially for uniqueness or formatting, to prevent user confusion.
5. **Nuxt SPA and Client-Side Auth**: When building a Nuxt app entirely behind a client-side authentication wall (like Firebase Client SDK), global SSR should be disabled (`ssr: false` in `nuxt.config.ts`). Attempting to SSR protected pages causes severe layout-breaking hydration mismatches when the client subsequently redirects to the login page.
6. **Full-Stack App Architecture**: Do not spawn subagents or use git worktrees when operating on a full-stack containerized app. Make changes directly in the main repository environment. Furthermore, ensure old CLI scripts and non-containerized code from previous iterations are completely removed to maintain a clean root directory.
7. **Bluesky API Search Constraints**: The Bluesky `app.bsky.feed.searchPosts` endpoint silently returns 0 results when an `OR` query (e.g. `AI OR Agent`) is combined with a `since` timestamp filter when sorting by `top`. To circumvent this, iterate over individual keywords and merge/deduplicate the results manually.
8. **Pydantic `None` vs Missing Keys**: When Pydantic models with `Optional` fields are serialized via `model_dump()`, fields default to `None` rather than being omitted. This means `dict.get("key", fallback)` returns `None` (not the fallback), because the key exists. Use `dict.get("key") or fallback` instead when a truthy default is required.
9. **Catalog Pattern for Firestore**: When multiple entities share a common definition (like data sources), use a top-level catalog collection with metadata and let user-specific configs reference it via a `source_id` string field. This avoids duplicating definitions across frontend/backend and makes the system data-driven instead of code-driven.
10. **Test Mocking for Firebase**: When `firebase_admin` is only installed inside Docker, mock it at the `sys.modules` level before any app imports in `conftest.py`. Use `patch.object` on the already-imported module rather than patching import paths.
11. **Pydantic Response Models Strip Extra Fields**: Pydantic response models in FastAPI will silently drop fields not defined in the model (e.g., an `existed` flag added dynamically). Design response models to include all fields you need, or use separate response models for different cases.
12. **Firestore Sub-Collection Migrations**: Renaming or moving data between collections/sub-collections in Firestore requires manual migration — there's no rename API. Plan for this in schema changes and document the old collection name for cleanup.
13. **Firestore Flat Collections vs Sub-collections**: When deciding between `users/{uid}/parent/{id}/child` and `users/{uid}/child` (with a `parent_id` field), lean towards a flat collection if read isolation is still achievable via indexes. Firestore bills reads identically for indexed queries, but flat collections are significantly easier to clean up via TTL (`expires_at`) without orphaned sub-collection documents that Firestore doesn't cascade-delete.
14. **Enablement-based Capping vs Creation-based Capping**: For multi-instance items (like subreddits or keywords), allowing uncontrolled creation but capping the number of **enabled** items provides better UX for curating configurations. Enforcement logic must reside in the update/toggle/bulk handlers, defaulting new additions over the limit to a disabled state.
15. **Local State Snapshot for Toggles**: Toggles subject to server-side rejection (e.g., tier limits) should capture an `originalState` before the API call and revert to it in the `catch` block to prevent the UI from desyncing with the real database state.
16. **Avoiding Nested Block Limits in Tests**: When tests require a large number of `patch.object` contexts (e.g., >20), Python will raise a `SyntaxError: too many statically nested blocks`. Use `contextlib.ExitStack` to manage these mocks dynamically and keep the code clean.
17. **Dynamic Configuration via Firestore**: Move application-wide constants (like tier limits) to a central `admin/config` Firestore document. This allows for real-time adjustments without code redeployments. Combine this with a `/api/users/me` endpoint to provide the frontend with authoritative, user-specific limits.

18. **Cloud Run Environment Variable Management**: While `gcloud` is the primary CLI for deployment, it does not natively support standard `.env` files. Instead, use the `--env-vars-file` flag with a **YAML** formatted file (`KEY: "VALUE"`). This is significantly more scalable and secure than manually passing comma-separated strings in the terminal.
19. **Dual-Layer Authentication for Cloud Run**: When deploying a frontend-backend pair, keep the Cloud Run services publicly accessible (`--allow-unauthenticated`) to simplify networking, but enforce strict **application-level authentication** (e.g., Firebase Auth) in the code. This provides the best balance between ease of deployment and robust data protection.
20. **Secure Builds via `.dockerignore`**: Always explicitly ignore `service-account.json` keys and `.env` files in `.dockerignore`. This prevents local development secrets from being accidentally baked into production Docker images, even if they aren't used by the application code at runtime.
21. **Containerized Execution Awareness**: Always respect the execution environment defined in `tasks/architecture_knowledge.md`. If the application is containerized via Docker Compose, **DO NOT** run ANY package-related commands (e.g., `npm run build`, `npm install`, `npx nuxi typecheck`, or `pip install`) directly on the host machine. These commands can corrupt the local environment and do not reflect the container state. Assume the containers are already running and interact with `localhost:3000` or `localhost:8000` directly.
22. **DaisyUI 5 Migration (btm-nav to dock)**: In DaisyUI 5+, the `btm-nav` component has been renamed to `dock`, and `btm-nav-label` is now `dock-label`. Using the old classes in a Tailwind v4 environment will result in standard block rendering (vertical stacking) and loss of fixed positioning/stickiness. Always use `dock` for bottom navigation in this project.
23. **Responsive Layout Breaking Elements**: Long, non-breaking strings (like raw URLs) or rigid flex headers can force a container's width beyond the mobile media query threshold during a desktop-to-mobile window resize. This prevents the mobile-only UI (like the `dock` navbar) from triggering. Always use `break-all`, `truncate`, and `flex-wrap` on potentially long content in responsive views.
24. **DaisyUI Loading States**: The `loading` class on a DaisyUI `btn` is often "destructive" — it replaces the button's internal text/icons with a spinner via a `:before` pseudo-element. For better UX, omit the `loading` class on the button itself and instead use an internal `<span class="loading loading-spinner"></span>` alongside the button text.
25. **OAuth Stale Promise Handling**: Firebase `signInWithPopup` promises can remain pending if a popup is closed but the SDK detection lags. If a user force-resets the UI state and tries again, a stale promise rejecting later can "hijack" the UI with an old error. Use an `attemptId` counter to invalidate results from previous attempts, and use `await nextTick()` + a short `setTimeout` delay before the next attempt to ensure the UI visually "flashes" and clears old error states.
26. **Expected Auth Error Suppression**: User actions like closing a sign-in popup throw `auth/popup-closed-by-user`. These are not system failures and should be caught at the composable level to prevent polluting the browser's developer console with red error traces, while still allowing the UI to handle the feedback gracefully.
27. **Reddit API 403 on Cloud Run**: When fetching from Reddit's unauthenticated JSON endpoints (like `/r/subreddit/rising.json`) from a Datacenter IP such as Google Cloud Run, default generic bot User-Agents will be met with a 403 Forbidden error. Emulate a standard, modern browser User-Agent to bypass these strict IP filters.
28. **Bypassing Reddit IP Blocks via Proxy/RapidAPI**: Even with browser emulation, Cloud Run IPs are often blanket-blocked by Reddit. Moving fetching logic to a specialized proxy service (like RapidAPI `reddit3.p.rapidapi.com`) provides a stable bridge. High-reliability parsers should implement this as the primary method with a direct request fallback for local development or non-blocked environments.
29. **API Key Security in Plan Mode**: When implementing new third-party integrations, never read the active `.env` files to avoid leaking secrets into the AI's internal reasoning or plan logs. Instead, use `.env.example` as the source of truth for key names and append placeholders to environment files.

30. **Firestore Security Rules and Client-Side Listeners**: Using the Firebase Web SDK `onSnapshot` listener directly in the frontend requires explicit **Firestore Security Rules**. Unlike the backend Service Account, the client SDK is subject to these rules. Without a proper error handler on the listener, permission denials will fail silently, causing UI elements (like status toggles or progress bars) to "hang" in a stale state. Always implement the error callback in `onSnapshot` to surface these issues.
31. **Robust Source Validation in Extractions**: When calculating sources used for an extraction, ensure that multi-instance sources (e.g., Reddit) are only included if they contain required parameters (e.g., a subreddit name). This prevents "phantom" sources from appearing in extraction metadata due to malformed or legacy documents in Firestore that might otherwise default to "enabled."
32. **Standardized UI for Tier Limits**: To ensure tier limits are consistently respected and displayed across different application flows (e.g., onboarding vs. settings), abstract limit warnings and counters into reusable components (like `UsageLimitBadge`). This guarantees the same presentation logic and reduces the risk of overlooking limit checks in new views.

33. **Atomic Quota Management in Firestore**: When implementing usage quotas (daily/weekly/monthly), use Firestore transactions for atomic incrementing to prevent race conditions. To handle the 15-day TTL of primary metadata, maintain a separate `quotas/` sub-collection for historical tracking. Frontend components must re-fetch the user profile after key actions (like starting an extraction) to keep UI counters in sync with the server-side state.
34. **Local vs Container Dev Dependencies**: Do not add local development tools (like `pre-commit`, `black`, `flake8`) to the main `requirements.txt` if they are not needed by the containerized production app (e.g., Cloud Run). Instead, install them locally within a virtual environment (e.g., `workon linkedin_posts`) to keep production images lean.

35. **Heuristic-based Categorization for Legacy Data**: When refactoring backend logic that categorizes data into types (e.g., `info` vs `warning`), existing database records might use the old, now "incorrect" labels. Frontend components should use content-based heuristics (message strings) alongside type labels to ensure consistent grouping and accurate categorization for both new and historical extraction data.
36. **Maintaining Solid Alert Aesthetics**: For critical or prioritized feedback (like zero-result warnings), users generally prefer standard, high-contrast "solid" alert styles. Avoid faint/pastel background overrides (e.g., `bg-opacity-10`) which can degrade readability and diminish the perceived importance of the notification.
37. **Responsive Badge Scaling**: When using badges in compact areas (like navbars or accordion headers), verbose labels (e.g., "12 entries") easily overflow on mobile. Use `hidden sm:inline` to hide descriptive text on small screens, leaving only the essential number or icon to preserve a clean UI layout.
